# 戦略スキル 執筆テンプレート

本スイートの `strat-*` スキルは、共通の節構成と 1 つの**出力契約**に従います。本書はその両方を定めます。執筆者は、節構成を必須の骨組みとして、出力契約を変更不可のインターフェース要件として扱ってください。Plan 3 の `acrosstudio-engagement` 統括スキルが、この契約に依存しています。

用語と見出しは `docs/authoring/strategy-glossary-ja.md`（日本語用語集）に揃えます。文体は `skills/acrosstudio-pptx/reference/content_patterns.md` の文体・禁止語に従います。

---

## 目的

戦略スキルは、再利用できるコンサルティングの「視点」です。各スキルは、ユーザーが述べた状況を受け取り、名前のついた分析フレームワークを当てはめ、単体で完結するマークダウンの分析を返します。これらの分析は連結して使う前提（あるスキルの成果物が次のスキルの入力になる）であるため、各成果物の形は予測可能で機械的に読み取れる必要があります。このテンプレートが、その形を担保します。

---

## 節構成

戦略スキルの `skills/<name>/SKILL.md` は、必ず次の節をこの順序で含めます。

```markdown
---
name: <skill-name>
description: 〜のときに使用。<このスキルが生成するものを一言で>。
---

# <フレームワーク名>

## 概要
このフレームワークが何であり、その中心となる分析の原則は何かを、1〜2 文で述べる。

## 使用する場面
- このフレームワークが必要になる状況や兆候。
- 使わない方がよい場面（別のフレームワークが適している、あるいはこれを使うと判断を誤る条件）。

## 必要な情報
このフレームワークが機能するために必要な情報。必須の情報が欠けている場合、スキルは必ずユーザーに尋ねる。数値・割合・顧客固有のデータを勝手に作ってはならない。

## 進め方
入力にフレームワークを当てはめる手順を番号付きで示す。前提知識のないコンサルタントでも追える程度に、具体的に書く。

## 成果物
このスキルは、下記の出力契約で定めた形の、単体で完結するマークダウン分析を生成する。

## 例
明確に架空とわかるシナリオを用いた簡潔な例（例：「例 — 架空の消費財メーカー」）。実在の顧客データを連想させてはならない。数値には「確認前」「例示」と添える。
```

### 執筆上の注意

- **`description` フィールド：**「〜のときに使用。」で始め、発動条件だけを述べる（進め方は書かない）。500 字以内を目安にする。`tests/skill-triggering/triggering.test.mjs` でトリガー語として使う語は、この `description` 内にそのまま含める。
- **`## 概要`：**一般的な定義ではなく、このフレームワークの目的と、それを有用にしている根底の着眼点を述べる。
- **`## 使用する場面`：**「使わない方がよい場面」を最低 1 つ含める。あらゆる状況を覆おうとするスキルより、正直に境界を引いたスキルの方が役に立つ。
- **`## 必要な情報`：**必須の入力を 1 つずつ挙げる。欠けている可能性があるものは「この数値が手元にない場合は、進める前にユーザーに確認する」と明記する。
- **`## 進め方`：**各手順に番号を振る。曖昧な動詞を避け、「コストを検討する」ではなく「コストを費目ごとに分けて列挙する」のように書く。
- **`## 例`：**架空の企業・施策を使い、明確にそうとわかるよう示す。フレームワークの働きが伝わる程度に具体化しつつ、短く保つ（20 行以内）。

## 出力契約

戦略スキルは、当てはめた結果として生成する分析を、必ず次の 2 つの節でこの順序で締めくくります。

```markdown
## 要点
## 確認事項
```

生成する成果物全体は、冒頭を次の形のトップレベルタイトルで始めます。

```markdown
# <フレームワーク> — <対象>
```

`<フレームワーク>` はスキルのフレームワーク名（用語集の日本語タイトル）、`<対象>` は分析対象となる具体的な案件テーマです（例：「現状分析 — Acme 社のデジタル変革」）。

### なぜ重要か

`acrosstudio-engagement` 統括スキル（Plan 3）は、各戦略スキルの成果物を受け取り、エンゲージメント全体の成果物へと連結します。統括スキルは `## 要点` の見出しを目印にしてスキルの境界を判別し、各スキルの確認事項を横断的に集めて、確認事項の一覧を作ります。スキルがこれらの節を省いたり、別の位置に置いたりすると、連結が壊れます。

冒頭の `# <フレームワーク> — <対象>` タイトルがあることで、統括スキルは合成文書の各節にラベルを付けられ、全体の成果物を読む人も素早く位置を把握できます。

### 契約の要点

| 要件 | ルール |
| --- | --- |
| 冒頭タイトル | 分析の最上部に `# <フレームワーク> — <対象>` |
| フレームワーク固有の本文 | フレームワークが必要とする各節を、スキルの `## 成果物` で定めた順序で並べる |
| 末尾から 2 番目の節 | `## 要点` — 最も重要な結論を短い箇条書きで |
| 最後の節 | `## 確認事項` — 顧客の確認、追加データ、後続の判断が必要な項目 |
| 作った数値 | 禁止 — 必要な数値が不明なら、確認事項として挙げる |

## 新しい戦略スキルをコミットする前のチェックリスト

- [ ] フロントマターの `name` がディレクトリ名と一致し、`description` が「〜のときに使用。」で始まっている。
- [ ] 6 つの節がすべて揃っている：概要、使用する場面、必要な情報、進め方、成果物、例。
- [ ] `## 成果物` の節が、出力契約の形（冒頭は `# <フレームワーク> — <対象>`、末尾は `## 要点` と `## 確認事項`）を説明している。
- [ ] `## 例` の節が、架空と明示したシナリオを使い、実在の顧客の数値を含んでいない。
- [ ] スキルのトリガー語が `tests/skill-triggering/triggering.test.mjs` に登録され、各語が `description` 内にそのまま現れている。
- [ ] `node scripts/validate-skills.mjs` がエラー 0 件を報告する。
- [ ] `npm test` が通る。
