# コンテンツルール詳細

Acrosstudio 資料の文章・コンテンツに関する詳細ルール。

## 1. 言葉遣い

### 基本姿勢
- 丁寧、誠実、押し付けがましくない
- 客先目線で書く（客先が読んで違和感ない）
- 専門用語は最小限、必要なら補足
- 「弊社」「貴社」 で統一

### 推奨表現

| シーン | 推奨 |
|---|---|
| 顧客企業を指す | 「貴社」 |
| 自社（Acrosstudio）を指す | 「弊社」 |
| 顧客企業の担当者を指す | 「担当の方」「ご担当者様」 |
| 提案する | 「ご提案します」 |
| 説明する | 「ご説明します」「ご紹介します」 |
| 検討する | 「ご検討ください」「ご相談ください」 |
| 確認する | 「ご確認ください」「お確かめください」 |
| 期待する効果 | 「期待される効果」「期待効果」 |
| 提案範囲 | 「ご提案範囲」「本提案範囲」 |

### 避ける表現

| 避ける | 理由 / 置換 |
|---|---|
| 「クライアント」 | 「貴社」 が望ましい |
| 「お客様」 | 「貴社」 の方がビジネス文書として一般的 |
| 「君」「あなた」 | 「ご担当者様」「貴社」 |
| 「我が社」 | 「弊社」 |
| 「うちは」「うちの」 | 「弊社では」「弊社の」 |
| 「絶対に」「確実に」 | 「確実に」 は実装の話、効果は「期待されます」 |
| 「劇的に」「圧倒的に」 | 誇大表現を避け、定量的に |

## 2. 禁止語と置換ルール

### 必ず置換する語

| 禁止 | 置換 | 理由 |
|---|---|---|
| **PoC**（Proof of Concept） | 「過去データで検証」「概念実証」 | 業界用語が客先に伝わらない |
| **ML / 機械学習** | 「AI モデル」「AI 活用」 | 一般用語化 |
| **アジャイル** | 「段階的に進める」「段階的開発」 | 業界用語 |
| **MLOps** | 「AI の運用基盤」 | 業界用語 |
| **エッジ AI** | 「装置側 AI」「現場側 AI」 | 業界用語 |
| **エンドツーエンド** | 「一気通貫」「業務全体」 | カタカナを避ける |

### ドメイン依存の置換例

| 業界 | 避ける | 推奨 |
|---|---|---|
| 半導体 | ロット停止 | 「品質ばらつき」「不良発生」 |
| 半導体 | 劣化 | 「経年変化」「状態変化」 |
| 製造 | チョコ停 | 「短時間停止」 |

### 客先固有名の抽象化

固有名（製品名・部品名・装置名・社内システム名）は提案書では抽象化する。以下は抽象化の要領を示す架空の例：

| 例（架空の製造装置）| 抽象化 |
|---|---|
| 型式 WX-1 | 「洗浄装置 A」 |
| 自動分類ユニット | 「自動分類システム」 |
| 品質管理ユニット | 「品質管理システム」 |
| 電源ユニット | 「電源装置」 |

理由：競合他社にも提案する可能性、社外秘の情報保護。

## 3. 文体の構造

### 文の長さ
- 1 文 50 字以内が目安
- 長い文は分割
- 1 段落は 3〜5 行程度

### 主語の明示
- 「弊社が」「貴社の」「AI が」 など主語を明示
- 受身表現は最小限（「〜される」「〜にされる」 を多用しない）

### 例：
- ❌「これにより業務が改善されると考えられます」
- ✅「これにより貴社の業務効率が向上します」

### 数字の書き方
- 数字 + 単位を明示（「12 ヶ月」「5,000 万円」 等）
- 範囲を示すときは「〜」 でつなぐ（「100〜200 件」）
- 桁区切り：3 桁ごとカンマ（「5,000,000 円」 or「500 万円」）
- 通貨：「円」 で統一

## 4. スライドのコンテンツ構造

### スライドタイトル
- 章番号付き：「1. 背景と過去の検討」「3.1 提案 1 ─ XXX」
- 章番号なし（パーツ資料など）：「01 故障予知保全 ─ 入力と出力の整理」

### リード文（サブメッセージ）
- スライドタイトルの直下
- そのスライドで「**何を伝えるか**」 を 1 文で
- 黒字（TEXT_DARK）、太字、14pt
- 例：「ベローズポンプの劣化予兆を検知し、計画外停止を計画化されたメンテナンスに転換します。」

### セクション見出し
- スライド内のセクション区切り（NAVY 色、太字、14〜16pt）
- 例：「内容」「入力」「出力」「期待される効果」「進め方」

### カードタイトル
- カード内の見出し（NAVY、太字、13pt）
- 短く、要点を表す
- 例：「受付業務の工数削減」「顧客トラブルの防止」

### カード本文
- 1〜2 文で説明
- 11pt、TEXT_DARK
- 「〜します」「〜できます」「〜に貢献します」 で締める

## 5. 提案書の標準構成

### フル提案書（最大 9〜10 枚）

1. **表紙**：タイトル、Ver、日付、白ロゴ
2. **背景・課題**：相談内容、過去の検討、業務フロー
3. **本提案の方向性**：アプローチ、新技術の説明
4. **提案 1 詳細**：内容、入力、出力、期待される効果
5. **提案 2 詳細**：同上
6. **提案 3 詳細**：同上
7. **概算金額感**：3 案の金額テーブル
8. **Appendix**：今後の展開候補
9. **進め方・次のステップ**：4 STEP の流れ

### 提案セクション（単独 1 枚）

- リード文（このセクションで何をするか）
- 内容（リード + 3 ポイント）
- 入力（4〜5 項目）/ 出力（3〜4 項目）の 2 カラム
- 期待される効果（3 カード）

### パーツ資料

| パーツ | 構成 |
|---|---|
| 入出力整理 | タイトル + ユーザー + リード文 + 3 カラム（入力/AI 処理/出力） |
| 運用費試算 | タイトル + 前提 + 2 カラム（クラウド/オンプレ）+ 注記 |
| ROI 試算 Excel | 投資コスト + 年間運用保守内訳 + 効果項目 + ROI 計算 |
| 概算金額感 | テーブル（案 / 概念実証 / 本開発 / 年間運用保守） |

## 6. フェーズ・段階の命名（固定ルールなし）

ロードマップやスケジュールで段階を示す場合、**固定の命名規則はない**。案件に応じた一般的な名称を使う：

- 例：**要件定義 → 開発 → 検証 → 運用**
- 例：**PHASE 1（要件定義/検証）→ PHASE 2（構築）→ PHASE 3（拡張）**

> 注：顧客によっては社内独自のフェーズ命名を使う場合があるが、これは**その顧客特有の呼称**であり、提案書全般で必須ではない。一般的な命名で問題ない。

### 見積もりの線引き（命名に関わらず共通）
- 段階ごとに金額を分け、「**どこまでが本提案範囲で、どこからが概算か**」を明示する。
- 要件定義と開発は別々に見積もるのが原則。

## 7. 効果の書き方

### 3 カード形式（推奨）

各カードに：
- **タイトル**（短い、12〜15 文字程度）：何が改善されるか
- **本文**（1〜2 文）：どのように改善されるか

例（提案 A：受領時の状態記録自動化）：

| タイトル | 本文 |
|---|---|
| 受付業務の工数削減 | VLM が状態を自然言語で自動記述し、担当の方は確認・修正のみ。手入力作業を大幅に削減します。 |
| 顧客トラブルの防止 | 受領時の状態を画像とテキストで自動的に証跡化。後の認識齟齬を未然に回避します。 |
| VLM 活用の基盤 | 提案 B・C への展開の足がかりとなる導入しやすい案。最短期間で効果を実感できます。 |

### 効果の書き方原則
- ❌「業務が効率化されます」（抽象的）
- ✅「受付業務の手入力工数が削減され、担当の方は判断業務に集中できます」（具体的）

- ❌「品質が向上します」
- ✅「担当者ごとの判断のばらつきを抑え、一定の基準で品質を確認できます」

### 数値効果の書き方
- 推測ベースなら明示：「想定で 30% 程度の工数削減」
- 実証されたら明示：「過去事例では 40% の削減実績」
- 範囲を示す：「20〜40% の改善が見込まれます」

## 8. 金額・期間の書き方

### 金額の表現
- 「概算」「規模感」「想定」 などの言葉で確定でないことを明示（「ざっくり」 等の砕けた表現は提案書では避ける）
- 「正式見積は要件定義完了後にご提示します」 を必ず添える
- 単位は「万円」 で統一（億円規模なら「億円」 もあり）

### 期間の表現
- 「約 X ヶ月」「X〜Y ヶ月」
- 「業務実態に応じて変動」 などの注記を添える

### 推測ベースの金額
- 「推測値です」「実際は変動します」 を明示
- 「装置 50 台規模を想定」 など前提を明示

## 9. 注記・補足の書き方

### 注記のスタイル
- フォント：11pt、GRAY_MID
- 「※」 で始める
- 1 行に収まる長さが望ましい

例：
- ※ 本資料の金額は仮置きの概算です。詳細はヒアリング後に正式見積をご提示します。
- ※ 装置 50 台規模・5 年保管・バッチ集約を前提。実態に応じて変動します。
- ※ 開発フェーズの正式見積は要件定義完了後にご提示します。

### 守秘義務関連
- 「Confidential」 表記をコピーライトに含める
- 客先固有情報は抽象化（前述）

## 10. 打合せ資料・議事録の書き方

### 議事録の構成
1. ヘッダー（日時、参加者、議題）
2. 要点・決定事項
3. 議論内容（論点ごとに）
4. 宿題・next action

### 検討資料の構成
1. 背景・目的
2. 現状認識・課題
3. 検討した選択肢（メリデメ比較）
4. 推奨案
5. リスク・留意点

### 報告書の構成
1. サマリ（1 行）
2. 詳細（章立て）
3. データ・根拠
4. 結論・next action

## 11. AI（Claude）が守るべき書き方原則

### ❌ 避けるべき
- 推測を「事実」 のように書く
- 無理に情報を盛り込む
- 専門用語を使いっぱなしにする
- 1 段落の長文（構造化されていない）
- 「絶対」「必ず」 などの断定（不確実なものを）
- 「劇的に」「圧倒的に」 等の誇大表現

### ✅ 推奨
- 推測なら「推測です」 と明示
- 必要十分な情報のみ
- 専門用語は補足（カッコ書き）
- 1 段落は 3〜5 行、要点ごとに改行
- 「期待されます」「想定されます」 で曖昧さを表現
- 数値や具体例で説得力を出す

## 12. ユーザー（Acrosstudio 社員）への質問のマナー

### 質問の原則
- 必要最小限に
- 一度に 1〜3 個まで
- 選択肢を提示（ユーザーが選びやすい）
- 推測で進めるなら明示してから確認

### 質問例
- ❌「どんな提案書を作りますか？」（漠然）
- ✅「対象は何業界の顧客でしょうか？ また、文書タイプは提案書 / パーツ資料 / 打合せ資料のどれですか？」（具体的）

## 13. サンプル資料から継承すべき暗黙ルール

### 単一テーマ提案書サンプルから
- ロット停止 → 自然な表現
- 劣化 → 自然な表現
- 装置・部品の固有名は抽象化
- 「過去データで検証」 を多用（PoC の代わり）

### フル提案書サンプルから
- リード文は黒字（TEXT_DARK）
- 効果は 3 カード（タイトル + 本文）
- 内容セクションは「リード + 3 ポイント」 構造
- フォントサイズ底上げ（本文 13、見出し 14〜16）
- 章番号付きタイトル（「1.」「3.1」 等）

### 共通
- 競合状況を踏まえ、弊社の強みが活きる領域を明確にした提案構成
- 「他社に譲るテーマ」 を明示する誠実さ
- 数字は推測ベースなら必ず明示

## 14. レビュー観点（ユーザーからの指摘でよくあるもの）

過去のフィードバックから学んだチェックポイント：

1. **フォントが小さい** → 本文 13pt 以上を基準に
2. **情報量が少ない** → 構造化されているか確認
3. **構造化されていない** → 1 段落の長文を避ける
4. **無理に盛りすぎ** → 推測ベースの情報を「事実」 のように書かない
5. **専門用語が伝わらない** → 客先目線で言い換え
6. **コピーライトとの重なり** → 視覚確認で必ずチェック
7. **数字の整合性** → 他資料との数字が食い違わないか
8. **言い回しが伝わらない** → 「03 から提供される」 ではなく「洗浄品質評価で構築される」 など内容で説明

これらは**作成時に予防**し、レビューで再度チェックする。

## 15. 競合戦略・優先度の表現

新規案件で「**どのテーマを取りに行くか**」 の優先度を客先に伝える際のパターン。

### 基本構造

```
ご提案対象の N 領域の内、どの領域でも対応可能であると考えております。

特に【本命テーマ】 は、【弊社の強み】 が直接活きる領域であり、
弊社の特徴が出るのではないかと考えております。
そして【次点テーマ】 は【本命テーマで蓄積される知見】 を活用し、
連携価値を発揮できる領域であると考えております。

なお、本優先度はあくまで弊社の特徴を踏まえたご参考であり、
いずれの領域につきましても柔軟にご対応いたします。
貴社のパートナー戦略・全体最適に応じてご相談ください。
```

### 構成要素

| 要素 | 役割 |
|---|---|
| 「どの領域でも対応可能」 | 全領域に対応できる前向きさを示す |
| 「特に X は」 | 本命テーマを明示 |
| 「弊社の特徴が出る」 | 控えめだが明確な強み訴求 |
| 「Y は X の知見を活用し」 | 次点テーマと本命との連携を示す |
| 「あくまで参考」 | 押し付けがましさを消す |
| 「柔軟にご対応」 | 全領域対応可能を再確認 |
| 「貴社のご判断」 | 最終判断は客先に委ねる |

### 「03 から提供」 ではなく「内容で説明」

❌ 避ける表現
- 「03 から提供される影響係数モデル」
- 「02 へ提供」
- 「テーマ 1 で構築されるもの」

✅ 推奨表現
- 「**洗浄品質評価で構築される**影響係数モデル」
- 「**プロセス調整提案で活用**される」
- 「**品質評価から得られる**条件と結果の関係性」

社内符号（番号・コード）を客先目線の**内容**に置き換える。

### 譲るテーマの伝え方

「他社に譲る」 を**ネガティブに見せない**書き方：

```
N 領域については、弊社の差別化が効きにくい領域です。
貴社のパートナー戦略・全体最適のご検討に応じて、
他のパートナー企業との協業もご選択いただけます。
```

## 16. 自社の強みの言語化

「**技術力**」「**実装力**」 のような抽象的な強みでは差別化にならない。**具体的に言語化**する。

### 良い例（具体的）
- 「**VLM**（Vision Language Model）」
- 「**FDE**（Forward Deployed Engineer）」
- 「**RAG**（Retrieval Augmented Generation）」
- 「**製造業ドメインの実装経験**」
- 「**エンタープライズ品質の運用設計**」

### 悪い例（抽象的）
- 「技術力」
- 「実装力」
- 「ノウハウ」
- 「経験」

### 強みを伝えるフォーマット

```
弊社は【具体的な強み】 に強みを持ちます。
これは【背景・実績】 で培った技術であり、
本案件の【特定のテーマ】 で直接活きると考えております。
```

## 17. Before / After で価値を見せる

提案の価値を伝えるには「**現状（Before）**」 と「**導入後（After）**」 の対比が効果的。

### パターン例

```
現状：故障してから対応する後手の保全
導入後：故障する前に計画的に対応する先手の保全

現状：担当者ごとに評価がばらつく目視判定
導入後：基準で揃った定量評価

現状：ベテラン頼みの判断
導入後：組織として持続可能な判断軸
```

### スライドへの落とし込み

- 左カラム：現状（GRAY_MID で控えめに）
- 右カラム：導入後（NAVY または MAGENTA で強調）
- 矢印：左 → 右
- カードで囲んで対比を明確に

## 18. 推奨パートナー戦略（提案の組み立て方）

複数テーマを扱う案件で「**全部やります**」 と「**1 つだけやります**」 の中間を狙う：

### 全部取りに行く戦略
- メリット：受注規模が大きい
- デメリット：弱い領域でも勝負することになる、信頼を損なうリスク

### 1 つだけに絞る戦略
- メリット：勝率が高い、専門性訴求
- デメリット：受注規模が小さい

### **2 つ取りに行く戦略**（推奨）
- 本命：差別化が明確な領域
- 次点：本命との連携で価値が広がる領域
- 残り：「対応可能」 と書きつつ譲る姿勢を見せる
- メリット：勝率と規模のバランス、誠実さも担保

### 提案書での表現

```
■ 推奨テーマ
1 位（本命）：◯◯（弊社の強み △△ が直接活きる）
2 位（次点）：××（本命との連携で価値拡大）

■ 対応可能テーマ
- ◇◇：弊社の技術力で対応可能、ご判断に応じてご相談ください
- □□：同上
```

## 19. 数字を客先に渡す際のマナー

### 推測ベースの数字
- 「**推測**」「**仮置き**」「**ご参考**」 を明示
- 「実態に応じて変動」 を必ず添える
- 前提条件（装置数・規模・期間）を明示

### レンジで示す
- ❌「500 万円」（断定的）
- ✅「**400〜600 万円**」（不確実性を含む）

### 比較・選択肢を示す
- 「クラウドなら X 万円、オンプレなら Y 万円」
- 「最小スタートなら X 万円、フル装備なら Y 万円」
- 客先が選びやすい構造に

### 数字を「いじれる」 形で渡す
- Excel なら入力セルを明確に（黄色背景）
- 「装置数」「人件費単価」 などのパラメータを変数化
- 客先が自分で試算できるように

## 20. 文書全体の組み立て（汎用テンプレ）

### 起承転結のフレーム

| パート | 内容 |
|---|---|
| **起**（背景） | 何が起きていて、なぜ検討しているか |
| **承**（課題） | 具体的な課題は何か |
| **転**（提案） | 弊社からの提案、新しいアプローチ |
| **結**（効果・進め方） | 期待される効果、次のステップ |

### 提案書なら 9〜10 枚に収める
- 1 枚で複数論点を詰めるより、**1 枚 1 論点**で見せる
- 余白があっても無理に埋めない

### 打合せ資料なら 4〜6 枚
- 議事録：表紙 + 要点 + 議論 + next action = 4 枚
- 検討資料：背景 + 課題 + 選択肢 + 推奨 = 4〜5 枚

## 21. 客先の判断を尊重する原則

提案は「**説得**」 ではなく「**判断材料の提供**」。

### やってはいけない
- ❌「これしかありません」 と選択肢を狭める
- ❌「絶対にこれが正しい」 と断定
- ❌ 客先の予算・状況を考慮しない
- ❌「他社では無理です」 と他を貶める

### やるべき
- ✅「**選択肢を整理してご提示します**」
- ✅「**弊社としては X を推奨します（理由付き）**」
- ✅「**最終的なご判断は貴社にお任せいたします**」
- ✅「**ご質問・ご相談はお気軽に**」

### 柔軟性を示す一文で締める
強い主張の後に**柔軟性を残す一文**を必ず添える。これで「押し付けがましさ」 が和らぎ、受け入れられやすくなる。
