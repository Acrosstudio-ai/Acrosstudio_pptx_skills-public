# Acrosstudio 資料作成スキル

Acrosstudio が顧客向けに作成する資料（提案書・パーツ資料・打合せ資料）を、統一されたデザイン・文体で作成するための指示書。

## このスキルを使う場面

ユーザーが「Acrosstudio の提案書を作って」「打合せ資料を準備して」「ROI 試算を」 等と依頼したら必ず参照する。対象は ①提案書 ②パーツ資料（入出力整理・運用費試算・ROI 試算・**お見積もりスライド**）③打合せ資料（議事録・論点整理・報告書）④既存 Acrosstudio 資料の修正。

## ⚡ ワークフロー（この順序で必ず進める）

資料作成を始めたら、**コードを書く前に** 必ずこの順序で進める。順序を飛ばすと過去に失敗している（章扉の独自創作・配色崩れ）。

### STEP 0a：環境を準備する（最初に・省略すると必ず失敗する）
ビルドを始める前に、ライブラリとアセットを用意する。これを飛ばすと「module not found」 や「ENOENT: assets/... が無い」 で確実に落ちる（過去に発生）。

```bash
# 1) ライブラリ（PPTX は Node、Excel は Python）
npm install pptxgenjs                       # PPTX 用
pip install openpyxl --break-system-packages # Excel 用（必要時）

# 2) 作業ディレクトリに assets/ をコピー（最重要）
#    example は "assets/logo_black.png" 等を相対パスで参照する。
#    ビルドを実行するディレクトリに assets/ が無いとロゴ・背景の読込でエラー停止する。
mkdir -p work/assets && cp /mnt/skills/user/acrosstudio-pptx/assets/* work/assets/
#    → work/ 内でビルドする（出力 .pptx も work/ に置く）
```

- **フォント**：本文は `BIZ UDPGothic`、英数字は `Arial`。環境に BIZ UDPGothic が無い場合は別フォントに静かに置き換わるため、**PDF 化（STEP 4）でフォントが反映されているか必ず目視確認**する。
- アセットの実体：`assets/logo_black.png`（右上）/`logo_white.png`（表紙・章扉右下）/`cover_texture.jpg`（表紙・章扉背景）。

### STEP 0：読む（最重要・省略禁止）
1. この SKILL.md の「絶対ルール」と「配色」を読む（下記）
2. **`reference/design_rules.md` を読む**（座標・マスター・章扉/アジェンダの仕様が全部ここにある）
3. **作る資料に最も近い `examples/*.js`（or `.py`）を開いて読む** → 自分でレイアウトを発明せず、これを土台に改変する

> 「要点だけ読んでコードを書き始める」 が一番の失敗パターン。example を開かずに章扉やカードを自作しない。

### STEP 1：情報を整理する
- 顧客企業（業界・規模・関係性）／文書タイプ／提案テーマ／スコープ（フル提案書 or 特定セクション）
- 不明点・未確定の数字は**推測で埋めず、ユーザーに確認**する

### STEP 2：構成を決める
- 文書タイプ別の標準構成は `reference/document_types.md` を参照
- 章扉・アジェンダを入れるかは「文書構造」 セクションの判断基準に従う（任意。勝手に独自デザインで作らない）

### STEP 3：ビルド
- pptxgenjs (Node.js) または openpyxl (Python) で、選んだ example を土台に作成
- `pres.layout = "LAYOUT_WIDE"`、マスターは example の定義をそのまま使う
- **必ず `work/`（assets/ をコピーした場所）で実行する**。example は `assets/...` を相対参照するため、別ディレクトリで実行するとロゴ・背景が読めず停止する

### STEP 3.5：PPTX を修復不要にする（省略禁止・PPTX のみ）

pptxgenjs は `[Content_Types].xml` に**実在しない `slideMaster{N}.xml` を指す `<Override>` をスライド枚数ぶん書き出す**バグがあり、そのままだと **PowerPoint で開くたびに「修復が必要」ダイアログが出る**（LibreOffice は寛容なので STEP 4 の PDF 化では気付けない）。ビルド直後・必ずサニタイズする。

```bash
# work/ 内（npm install pptxgenjs 済み）で、生成した .pptx ごとに実行する
node /mnt/skills/user/acrosstudio-pptx/scripts/sanitize_pptx.mjs "{file}.pptx"
# → "removed N phantom slideMaster override(s)" と表示されれば修復不要な状態
```

- jszip は pptxgenjs の依存として入るため追加インストール不要。**必ず `work/` から実行**する（CWD の node_modules から jszip を解決する）。
- Excel（openpyxl）はこの問題の対象外。PPTX（pptxgenjs 出力）のみ実行する。

### STEP 4：視覚確認（省略禁止）
```bash
# ファイル名は必ずダブルクオートで囲む（日本語ファイル名は無クオートだと失敗する）
python3 /mnt/skills/public/pptx/scripts/office/soffice.py --headless --convert-to pdf "{file}.pptx"
pdftoppm -jpeg -r 110 "{file}.pdf" slide
# 生成された slide-*.jpg を view ツールで 1 枚ずつ確認する
```
確認：配色規律・フォントサイズ・はみ出し・ロゴ/コピーライトと本文の重なり・レイアウトの乱れ。問題があれば修正して STEP 3〜4 を繰り返す。

### STEP 5：納品（ユーザーに渡す）
- **PPTX は STEP 3.5 のサニタイズ済みファイルを渡す**（未サニタイズのまま納品しない）
- 完成した `.pptx`／`.xlsx` を `/mnt/user-data/outputs/` にコピーし、present_files で提示する
- ファイル名は命名規約（後述）に従う

## 🚫 絶対ルール（やってはいけないこと）

| 禁止 | 正しい対応 |
|---|---|
| **色帯（rect）の上にテキストを別オブジェクトで重ねる** | テキスト 1 個に `fill`＋`line`＋必要なら `shape:"roundRect"`／`inset` を直接指定して**単一オブジェクト**にする。座標を二重管理しない（下記「オブジェクト設計」） |
| **スキル/example にない独自スライドや装飾を発明する**（例：ネイビー一色＋マゼンタ下線の章扉） | 章扉を入れるなら**表紙と同じテクスチャ背景**にする。`reference/design_rules.md §5.5` と `examples/proposal_full.js` の `buildDivider` を踏襲 |
| **MAGENTA を汎用アクセントに使う**（矢印・章番号下線・「追加」帯・ロードマップの一部フェーズ等） | MAGENTA は「対比」と「効果」専用（下記）。それ以外は **NAVY**。迷ったら NAVY |
| **CLIENT をスライド要素の色付けに使う**（カードヘッダー・入力帯・比較元帯など） | CLIENT は顧客コーポレートカラーのフォールバック定数であり、スライド上の帯・ヘッダー・アクセントには一切使わない。すべて **NAVY** |
| **顧客カラーを地（タイトル・見出し）に使う** | 地は必ず NAVY。顧客コーポレートカラーはロゴ横など小面積のアクセント限定 |
| **特定顧客の符号・命名を流用する**（製品名、独自フェーズ名など） | 一般名称（要件定義・開発・検証・運用 等）に置換。`reference/content_patterns.md` 参照 |
| **禁止語をそのまま使う**（PoC・ML 等） | 「概念実証」「AI モデル」 等に置換（下記の表） |
| **推測ベースの数字を事実のように書く** | 「想定」「概算」「ご参考」 と明示、レンジ＋前提を添える |
| **視覚確認（STEP 4）を省略する** | コードが正しくても見栄えが悪ければ NG。必ず PDF 化して目視 |
| **PPTX のサニタイズ（STEP 3.5）を省略して納品する** | pptxgenjs 出力は必ず `scripts/sanitize_pptx.mjs` を通す。省略すると PowerPoint で「修復が必要」ダイアログが毎回出る。LibreOffice の PDF 化では検知できないため STEP 4 とは別に必須 |
| **スライド端に要素を寄せる／マージンを無視する** | すべての要素（カード・テキスト・帯・矢印）は左端 `x ≥ LAYOUT.MARGIN（0.533）`、右端 `x + w ≤ 13.33 - LAYOUT.MARGIN（12.797）`、本文開始 `y ≥ LAYOUT.BODY_TOP（1.95）`、下端 `y + h ≤ LAYOUT.COPYRIGHT_TOP（7.00）` を厳守。LAYOUT 定数を使って算出し、ベタ書きした `x: 0` や `x: 0.1` などマージン割れの座標を書いてはならない |
| **縦方向の積み上げ計算をコード前に行わず下端を超過する** | 複数セクションを縦に並べるスライド（提案本文・まとめ等）では、**コードを書く前に**各セクションの高さを合計して `LAYOUT.COPYRIGHT_TOP（7.00）` に収まるか必ず紙上計算する。計算式：`LAYOUT.BODY_TOP + Σ(各セクション高さ + 余白) ≤ 7.00`。超過する場合はセクション数を減らすか、各カード・ポイント行の高さを縮小してから実装を始める。`LAYOUT.COPYRIGHT_TOP` を定義するだけで実際の座標計算に使わないのは「定義しただけ」であり違反と同等。コード記述後も `最終要素の y + h` の数値を明示して `≤ 7.00` を確認すること |
| **LAYOUT 定数を定義せずに座標をベタ書きする** | 必ずスクリプト冒頭に `const LAYOUT = { MARGIN: 0.533, CONTENT_W: 12.264, ... }` を定義し、すべての x/y/w をそこから算出する。マジックナンバー（例：`x: 2.90`）の散在は禁止 |
| **`addText` に `fill` を指定して中央揃えを期待する** | `addText` に `fill` を付けると `align:"center"` が機能しない（pptxgenjs の挙動）。**枠は `addShape("rect")` で描き、テキストは `fill` なしの `addText` を重ねる**（rect と text の分離）。お見積もりの値セルはこのパターンで実装する |
| **runs 形式（配列）の `addText` で外側の `align` を信頼する** | runs 形式では外側の `align` が各 run に引き継がれない。各 run の `options` に個別に `align:"center"` を明示する |
| **表紙テキストを NAVY や TEXT_DARK で書く** | テクスチャ背景は暗色のため、表紙のタイトル・サブタイトル・Ver/日付はすべて **WHITE** で書く |
| **右上ロゴのサイズ・位置を変更する** | 黒ロゴ（通常スライド右上）のサイズと位置は以下の値に固定する。変更禁止。`x: 10.427, y: 0.10, w: 2.903, h: 0.792`（単位：inch）。**`examples/` のコードに別の値が書かれていても絶対ルールを優先すること**（examples は土台用サンプルであり、この値は誤りを含む場合がある） |
| **NAVY とパープル以外の色を使う** | 使用できる色は **NAVY**（主色）と**パープル**（強調専用）の 2 色のみ。CLIENT・MAGENTA・GRAY 系・NAVY_LIGHT 等その他の色を帯・ヘッダー・カード枠・アクセント等に使うことは禁止 |
| **パープルを強調（結論）以外に使う** | パープルは「結論・最重要強調箇所」にのみ使用可。1 つの資料（ファイル）内で使用できる箇所は **1 箇所のみ**に制限する。2 箇所以上にパープルを使うことは禁止。複数の候補がある場合は最も重要な 1 箇所を選んで使い、残りは NAVY にする |

## 配色

```
NAVY        #073763   主色：タイトル・見出し・章番号・カード枠・矢印・進行系
NAVY_LIGHT  #DCE6F0   背景帯・リード文枠・アジェンダ現在地
CLIENT      #2C7DA6   顧客カラーのフォールバック専用（顧客指定色が未定の場合のみ）
MAGENTA     #A22B94   ★対比/効果のみ（下記）
MAGENTA_LIGHT #F4E4F0 使用禁止（対比・効果以外のいかなる用途にも使わない）
GRAY_MID    #595959   補足・注記・コピーライト
GRAY_LINE   #BFBFBF   罫線・カード枠
TEXT_DARK   #262626   本文
WHITE       #FFFFFF   背景・表紙テキスト
```

### MAGENTA を使ってよいのは次の 2 つだけ
1. 従来 ↔ 提案／Before ↔ After の対比における**提案・After 側**（比較元の現状側は NAVY）
2. 「期待される効果」3 カードの**左アクセントバー**

→ これ以外は NAVY。MAGENTA を置く前に「これは After 側か効果か？」を自問する。

### MAGENTA の厳格な使用判定 ★違反が繰り返し発生

`MAGENTA`（紫）は**After 側または効果カードの左バーのみ**使う。`CLIENT`（青）はスライド上の要素の色付けには使わない（顧客コーポレートカラーの代替定数としてのみ存在する）。コードを書く前に下記の判定フローを必ず通ること。

#### 判定フロー

```
色を使おうとしている要素は何か？
│
├─ Before↔After 対比スライドの After・提案側
│   └─ → MAGENTA を使ってよい（Before・現状側は NAVY）
│
├─ 「期待される効果」カードの左アクセントバー
│   └─ → MAGENTA を使ってよい
│
└─ 上記以外のすべて（顧客側要素・入力カード・アジェンダ・ステップ・まとめ等すべて）
    └─ → NAVY 一択。CLIENT・MAGENTA は禁止
```

#### 過去に実際に起きた違反例（実案件）

| スライド | 違反内容 | 正しい色 |
|---|---|---|
| 「なぜ今AIなのか」左ヘッダー | `CLIENT` で「市場環境の変化」を着色 → Acrosstudio の説明コンテンツなので対比でない | `NAVY` |
| 「なぜ今AIなのか」右ヘッダー | `MAGENTA` で「今こそ行動すべき理由」を着色 → Before↔After でなく提案説明 | `NAVY` |
| 「AIが変える3つの業務領域」01カード | `CLIENT` で「業務効率化」を着色 → Acrosstudio の提供コンテンツ | `NAVY` |
| 「AIが変える3つの業務領域」03カード | `MAGENTA` で「顧客体験の向上」を着色 → 効果カードの左バーでなくヘッダー全体 | `NAVY` |
| 「まとめ・ネクストアクション」01・03カード | `CLIENT`/`MAGENTA` でカードを色分け → ネクストアクションは対比でない | `NAVY` |

> **原則：迷ったら NAVY。** `CLIENT` や `MAGENTA` を使う積極的な理由（対比・効果の明示）がなければ、常に NAVY を選ぶ。

### カード本文エリアの背景色 ★過去に違反が発生

カードのヘッダー帯（`CLIENT` / `MAGENTA` / `NAVY`）に対応する薄色（`CLIENT_LIGHT` / `MAGENTA_LIGHT` / `NAVY_LIGHT`）を本文エリアの背景に使いたくなるが、**これは禁止**。

| 色 | 許可される用途 | ❌ やってはいけない用途 |
|---|---|---|
| `CLIENT_LIGHT` `#D9E5EC` | （単独では使わない） | カード本文エリアの背景 |
| `MAGENTA_LIGHT` `#F4E4F0` | （単独では使わない） | カード本文エリアの背景・支払条件ボックス・注意喚起ボックス等あらゆる用途 |
| `NAVY_LIGHT` `#DCE6F0` | リード文枠・アジェンダ現在地ハイライト | カード本文エリアの汎用背景 |

**カード本文エリアの背景は必ず `WHITE`（`#FFFFFF`）**。枠線は `GRAY_LINE`（`#BFBFBF`）を使う。

```javascript
// ✅ 正しい：本文エリアは WHITE + GRAY_LINE
s.addShape("rect", {
  x, y: headerY + headerH, w, h: bodyH,
  fill: { color: C.WHITE }, line: { color: C.GRAY_LINE, width: 0.7 },
});

// ❌ 誤り：ヘッダー色に対応する薄色を本文背景に使う
s.addShape("rect", {
  x, y: headerY + headerH, w, h: bodyH,
  fill: { color: C.CLIENT_LIGHT },  // ← 禁止
  line: { color: C.CLIENT, width: 0.7 },
});
```

### 表紙テキストの色 ★同化に注意

表紙は暗いテクスチャ背景（`cover_texture.jpg`）を使うため、テキストを NAVY や TEXT_DARK で書くと**背景と同化して読めなくなる**。表紙上のすべてのテキスト（タイトル・サブタイトル・Ver/日付）は必ず `WHITE` を使う。

```javascript
// ✅ 正しい：表紙テキストはすべて WHITE
s.addText("AIで変わる、これからのビジネス", {
  color: C.WHITE, fontSize: 36, bold: true, ...
});

// ❌ 誤り：NAVY や TEXT_DARK は暗い背景に同化する
s.addText("タイトル", { color: C.NAVY, ... });   // 読めない
s.addText("サブタイトル", { color: C.TEXT_DARK, ... }); // 読めない
```

### 顧客コーポレートカラー
提案書は Acrosstudio が出す資料。**地（タイトル・見出し・本文）は NAVY を維持**し、顧客指定のコーポレートカラーがある場合はロゴ横や小さなアクセント要素に限定して使う（`CLIENT` 定数はそのフォールバック値として存在するに過ぎない）。
- **使わない**：タイトル・章番号・カードヘッダー・入力カード・比較元側の帯・Acrosstudio の提供物・ロゴ/コピーライト
- **取得**：①ユーザー指定 → ②`web_search`（"{企業名} brand color hex" / ロゴ・ブランドガイド）→ ③不明なら**ユーザーに確認**（推測で確定しない）
- **可読性**：明るい/淡い色は白文字が乗らない。帯に使うなら濃いトーン、無理なら細枠・左バーなど小面積に。**必ず PDF 化して可読性を視覚確認**
- **定数化**：`const CLIENT_BRAND = "XXXXXX";`（未指定時は `CLIENT` にフォールバック）。詳細は `reference/design_rules.md §1`

- 日本語 **BIZ UDPGothic** ／ 英数字 **Arial**
- スライドサイズ **LAYOUT_WIDE**（13.33 × 7.5 inch）

| 用途 | サイズ |
|---|---|
| スライドタイトル（章番号付き含む） | 22pt |
| セクション見出し | 14〜16pt |
| カードタイトル | 13〜14pt |
| **リード文** | **14pt（黒字 TEXT_DARK・太字）** |
| 本文 | 11〜13pt |
| 注記 | 10〜11pt（GRAY_MID） |
| コピーライト | 8pt（Arial・GRAY_MID） |

### マスター要素（全通常スライド共通）
- **黒ロゴ**：右上 `x:10.427, y:0.10, w:2.903, h:0.792`（絶対ルール固定値。変更禁止）
- **コピーライト**：右下 `x:7.332, y:7.092, w:5.599, h:0.293`、`Copyright {YYYY}. Acrosstudio, Inc. <Confidential>`、Arial 8pt GRAY_MID 右揃え
- **表紙・章扉はこれを入れない**（テクスチャ背景＋白ロゴの別マスター）

## オブジェクト設計（メンテナンス性）

スライドのオブジェクト数は最小にする。同じ見た目なら、座標を二重管理しない作りを選ぶ。

### 色帯付きテキストは「単一オブジェクト」で作る
色のついた帯・バッジ・ボタンの上に文字を載せる場合、**rect と text を重ねない**。テキスト 1 個に背景色・枠線・形状を直接指定する。pptxgenjs では `addText` が `fill` / `line` / `shape` / `inset` を受け付ける（環境で描画確認済み）。

```javascript
// ❌ 旧：2 オブジェクト（rect と text を座標合わせで重ねる）
s.addShape("rect", { x, y, w, h: 0.48, fill: { color: C.NAVY }, line: { type: "none" } });
s.addText(text, { x: x + 0.2, y, w: w - 0.4, h: 0.48, color: C.WHITE, ... }); // ←座標を別管理

// ✅ 新：1 オブジェクト（text に fill+inset を直接）
s.addText(text, {
  x, y, w, h: 0.48, color: C.WHITE, bold: true, valign: "middle",
  fill: { color: C.NAVY }, line: { type: "none" }, inset: 0.2,
});
```

- 角丸が必要なら `shape: "roundRect", rectRadius: 0.05` をテキストに付ける（rect 不要）。
- 左寄せの帯は `inset`（インチ）で内側余白をとる。`margin` は効かないので使わない。
- これで座標の二重管理が消え、サイズ変更・色変更が 1 箇所で済む。

### inset（内側余白）の基準 ★デザイン品質に直結
色帯に直接テキストを置くと、**左寄せの場合は文字が帯の端にぴったり付いて不格好**になる。必ず内側余白をとる（描画確認済みの推奨値）：

| 状況 | inset | 備考 |
|---|---|---|
| 左寄せの色帯・見出し（`align:"left"`） | **0.15〜0.20** | 0 は端に貼り付き NG。0.15〜0.20 で適切な余白 |
| 中央寄せ（`align:"center"`、番号バッジ等） | 0 で可 | 中央寄せは端に付かないため余白不要 |
| 本文を内包する塗り枠（注記ボックス等） | 0.15〜0.25 | 上下も詰まらないよう `valign` と併用 |

> ルール：**左寄せで色背景を付けるテキストには必ず `inset: 0.15` 以上**。中央寄せは不要。迷ったら 0.2。

### 要素間の縦方向スペース（Y軸余白）★間詰まり防止

カード内に複数の要素（バッジ・ラベル・タイトル・期間・本文）を縦に並べるとき、隣接要素の `y + h` と次要素の `y` の差（＝余白）が小さすぎると窮屈に見える。過去の修正履歴から得た推奨値：

| 隣接する要素の組み合わせ | 推奨余白 | 備考 |
|---|---|---|
| 浮きバッジ（楕円）下端 → カード上端 | **0.20 以上** | バッジ `y + h` とカード `y` の差。0.10 では密着して見える |
| カード上端 → 最初のラベルテキスト | **0.60 以上** | バッジがカードに半分かかる構造の場合、ラベル開始を深めに |
| STEPラベル → タイトル | **0.40 以上** | 小さいラベルと大きいタイトルの間は広めに |
| タイトル → 期間テキスト | **0.45 以上** | サイズ差があるほど詰まって見えるため注意 |
| 期間テキスト → 本文 | **0.20 以上** | 区切りが不明瞭になるため必ず余白を入れる |
| 左右余白（本文 x offset） | **0.28 以上** | 0.18 では端に近すぎる。カード幅が狭い（3inch未満）場合は特に重要 |

**ステップカードの実装例（推奨座標）：**

```javascript
const startY = 2.30;   // カード上端（バッジは startY - 0.40 に配置）
// バッジ下端 = startY + 0.40、リード文との間を確保するため startY を十分に下げる

s.addText(step.num,    { y: startY + 0.65, h: 0.40 }); // バッジ下端から +0.25 の余白
s.addText(step.title,  { y: startY + 1.10, h: 0.50 }); // STEPラベルから +0.45
s.addText(step.period, { y: startY + 1.60, h: 0.35 }); // タイトルから +0.50
s.addText(step.body,   { y: startY + 2.20, h: 1.25 }); // 期間から +0.25 の余白
```

> フォントサイズを変えて余白不足を補おうとしない。サイズは基準値を守り、**Y座標を動かして余白を作る**のが正しい対処。

### 座標・寸法は定数で管理する（省略・ベタ書き禁止）

マージンや段組み幅を各所にベタ書きしない。**スクリプト冒頭に必ず LAYOUT 定数を定義**し、全要素がそれを参照する。ベタ書きの座標値（マジックナンバー）は禁止。

```javascript
// ── レイアウト定数（全スライド共通。ベタ書き厳禁・省略禁止）──
const LAYOUT = {
  MARGIN:         0.533,   // 左右マージン（すべての要素の x はここ以上）
  CONTENT_W:     12.264,   // コンテンツ幅 = 13.33 - MARGIN*2
  RIGHT_EDGE:    12.797,   // 右端の死守線 = 13.33 - MARGIN（x+w はここ以下）
  TITLE_Y:        0.333,   // タイトル上端
  LEAD_Y:         1.226,   // リード文上端
  BODY_TOP:       1.95,    // 本文・カード開始の最小 Y
  BOTTOM_EDGE:    7.00,    // 下端の死守線（y+h はここ以下）
  COPYRIGHT_TOP:  7.092,   // コピーライト帯の上端（本文をここに重ねない）
  GAP:            0.15,    // カード間ギャップ（最小値）
  INSET:          0.18,    // 左寄せ色帯の標準内側余白
};
// ── カラム幅の算出例（常に LAYOUT 定数から導出する）──
// 3 カラム：const colW = (LAYOUT.CONTENT_W - LAYOUT.GAP * 2) / 3;
// 2 カラム：const colW = (LAYOUT.CONTENT_W - LAYOUT.GAP) / 2;
// カードの x：const x = LAYOUT.MARGIN + i * (colW + LAYOUT.GAP);
```

#### マージン違反チェック（コード生成後・必ず自己確認）

コードを書いたら、全 `addShape` / `addText` / `addImage` の座標を次の 4 条件で確認する。1 つでも違反があれば修正してからビルドする。

| チェック | 条件 | NG 例 |
|---|---|---|
| **左マージン** | `x >= 0.533` | `x: 0`、`x: 0.1`、`x: 0.3` |
| **右マージン** | `x + w <= 12.797` | `w: 13.33`（全幅）、`x: 0.533, w: 12.8` |
| **上マージン（本文）** | カード・本文の `y >= 1.95` | タイトル直下にカードを `y: 1.0` で置く |
| **下端** | `y + h <= 7.00` | `y: 6.5, h: 0.8`（= 7.3 で超過） |

> Before/After フローなどで横幅を広く取りたい場合も `x` は必ず `LAYOUT.MARGIN` から始め、`w` は `LAYOUT.CONTENT_W` 以内に収める。

## テキスト溢れ・はみ出し対策

過去に、注記やテーブルで文字が枠外へ漏れる／フッター（コピーライト）に重なる事故が起きている。次を守る。

- **下端の死守線**：本文・図形は `y + h ≤ 7.0`（インチ）に収める。コピーライト帯（7.09〜）に重ねない。
- **入る文字量の目安**（BIZ UDPGothic、幅 inch あたり）：12pt ≒ 全角 4.3 文字/inch、13pt ≒ 4.0、14pt ≒ 3.7。例：幅 4.0inch・13pt の 1 行は全角 16 文字前後が上限。超える見込みなら改行されるので `h` を行数ぶん確保する。
- **収まらない時の優先順位**：①**文言を削る・要約する**（最優先。提案書は簡潔が善）→ ②改行して `h` を増やす → ③列幅・配置を見直す → ④それでも無理なら最後にフォント -1pt。**いきなり文字を小さくしない**（基準サイズを下回ると可読性・統一感が崩れる）。
- **テーブル**：行数 × 行高 + ヘッダー高 + 注記高 がフッター線を超えないか、置く前に概算する。
- **最終確認は目視**：上記は目安。STEP 4 の PDF 画像確認で、はみ出し・重なり・余白の不均衡を必ずチェックする。

### 要素重なり防止（★繰り返し発生する問題）

以下の 4 パターンは実際に繰り返し発生した重なり問題。コード記述後・ビルド前に必ず数値チェックする。

#### ① 注記・補足テキスト ↔ 直上のカード・図形

```javascript
// ✅ 正しいパターン：endY 変数を使って注記位置を算出
const cardEndY = cardY + cardH;          // 直上要素の下端
const noteY    = cardEndY + 0.20;        // 余白 0.20 以上を確保
s.addText("※ 注記...", { x: LAYOUT.MARGIN, y: noteY, ... });

// ❌ 誤り：固定値で注記を置き、カードと重なった
s.addText("※ 注記...", { x: LAYOUT.MARGIN, y: 6.65, ... });
// → カードが予想より大きかったとき 6.65 がカード内に入り込む
```

**セルフチェック**：コードを書いたら `noteY >= cardEndY + 0.15` を数値で確認する。PDF でも注記が枠内に入り込んでいないか確認する。

#### ② ステップ間・フロー間の矢印 ↔ 隣接カード

```javascript
// ✅ 正しいパターン：矢印は gap の中央に収める
const gap      = 0.20;                   // カード間の隙間（inches）
const arrowW   = 0.15;                   // 矢印の幅
const arrowX   = cardX + cardW + (gap - arrowW) / 2;  // gap 内の中央
// 確認: arrowX + arrowW <= nextCardX
s.addText("▶", { x: arrowX, y: arrowY, w: arrowW, ... });

// ❌ 誤り：矢印を cardX + cardW に直置きして次カードに食い込んだ
s.addText("▶", { x: cardX + cardW, y: arrowY, w: 0.15, ... });
```

**セルフチェック**：`矢印x + 矢印w <= 次カードのx` を数値で確認する。

#### ③ 縦並びセクション ↔ 縦並びセクション（ヘッダー帯 + カード群）

```javascript
// ✅ 正しいパターン：セクション終端を変数で追跡する
const sec1HeaderY = LAYOUT.BODY_TOP;
const sec1HeaderH = 0.45;
const sec1CardH   = 2.00;
const sec1EndY    = sec1HeaderY + sec1HeaderH + sec1CardH;  // 上セクション終端

const SECTION_GAP = 0.30;
const sec2Y = sec1EndY + SECTION_GAP;   // ← 変数から算出。固定値で書かない

// ❌ 誤り：固定値で下セクションを置いて上と重なった
const sec2Y = 3.80;   // → 上セクションが 4.10 まであればがっつり重なる
```

**セルフチェック**：スライドに複数セクションがある場合、各セクションの `startY` と `endY` を紙か変数コメントに書き出し、`sec2Y > sec1EndY + 余白` を数値確認してからビルドする。

#### ④ ガントチャート・スケジュールの右端

```javascript
// ✅ 正しいパターン：コード記述前に右端を計算する
const labelW  = 3.00;   // 左ラベル列の幅
const months  = 12;
const RIGHT   = LAYOUT.MARGIN + LAYOUT.CONTENT_W;  // = 12.797
const cellW   = (RIGHT - LAYOUT.MARGIN - labelW) / months;
//            = (12.797 - 0.533 - 3.00) / 12 = 0.772  ← 自動計算でマージン厳守
const startX  = LAYOUT.MARGIN + labelW;

// ❌ 誤り：cellW を先に決めてはみ出した
const cellW  = 0.95;    // 0.533 + 3.00 + 12×0.95 = 14.933 → 右端超過
```

**セルフチェック**：`LAYOUT.MARGIN + labelW + 月数 × cellW <= 12.797` を必ず計算してからビルドする。PDF でガント右端がスライドに収まっているか確認する。

### rect（addShape）を使ってよい場合
- **文字を載せない純粋な装飾**（タイトル左の縦アクセントバー、区切り線、矢印 `rightTriangle` 等）
- **複数の子要素を内包するカード枠**（枠の中に別座標で複数テキスト・帯を置くコンテナ。枠自体は文字を持たない）

→ 「その rect は文字を載せるためだけにあるか？」を自問。Yes なら text の `fill` に統合する。

### 標準構成
文書タイプ別の構成は `reference/document_types.md` を参照。提案書フルなら 表紙 → 背景/課題 → 方向性 → 提案詳細 → 金額 → 進め方 が基本。

### 章扉・アジェンダ（どちらも任意）
- **判断基準**：枚数が多い／章が明確に分かれる提案書では入れると親切。短い提案やパーツ資料では不要。**ユーザー指示があればそれに従う。指示がなければ付けても付けなくてもよいが、付ける場合は下記仕様を厳守**。
- **章扉**：表紙と同じテクスチャ背景＋章番号・章タイトル（白）＋白ロゴ右下。独自背景・装飾は禁止。→ `examples/proposal_full.js` の `buildDivider`
- **アジェンダ**：通常マスター上に NAVY 番号バッジ＋章カード（現在地は NAVY_LIGHT）。MAGENTA 不使用。→ 同 `buildAgenda`

ロードマップやスケジュールのフェーズ名は、**案件に応じた一般名称**（要件定義・検証・開発・実装・運用 等）を使う。特定顧客で使われていた固定命名を流用しない。固定のフェーズ体系を強制する必要はない。

### コンテンツの構造化
- 各スライドは「**リード文（14pt 黒太字）＋ 構造化ポイント**」 で構成
- 1 段落の長文を避け、見出し付きカード・3 カラム・箇条書きを多用
- 効果や特徴は **3 カード形式（タイトル＋説明）**

## コンテンツ規約

### 言葉遣い
- 「貴社」「担当の方」「弊社」 で統一。「ご相談」「ご提示」「ご検討」 等の丁寧表現
- 客先固有名（製品名・部品名・装置名）は**抽象化**（例：型式 WX-1 →「装置 A」）
- 社内符号は使わない（「03 から提供」 → 「洗浄品質評価で構築されたもの」 と内容で説明）

### 禁止語と置換
| 禁止 | 置換 |
|---|---|
| PoC | 「概念実証」「過去データで検証」 |
| ML / 機械学習 | 「AI モデル」 |
| ロット停止 / 劣化 | 自然な表現で言い換え |

### 数字・通貨のフォーマット（資料間で統一）
- **通貨単位は「万円」 で統一**（億規模なら「億円」）。例：`500 万円`、`1,500〜2,500 万円`
- **3 桁区切りカンマ**：`5,000,000 円` または `500 万円`
- **パーセント**：`0.0%` 書式（例：`30.0%`）。本文中は「30% 程度」 でも可
- **年・期間**：`5 年`、`3〜6 ヶ月`。Excel セルは `0.0"年"` 書式
- **レンジは「〜」 でつなぐ**：`100〜200 万円`、`3〜6 ヶ月`
- 同一案件で複数資料（提案書・ROI Excel・運用費試算）を作る場合、**同じ数字・同じ単位・同じ書式**にそろえる（食い違いは指摘されやすい）

### 「たたき台」運用の作法
初期提案は確定情報が少ない。未確定を誠実に示すことで信頼を得る。
- **表紙・冒頭に位置づけを明示**：「Ver 0.x ／ 議論用たたき台」「複数回の打ち合わせで解像度を上げる前提」
- **未確定の数字は必ずラベル付け**：`想定` / `概算` / `ご参考` / `要協議`。確定値と混在させない
- **空欄を推測で埋めない**：情報がなければ「要協議」「ヒアリング後にご提示」 と書く（無理に盛らない）
- **前提を併記**：「装置 50 台規模を想定」「5 年保管前提」 等、数字の根拠を添える

### ファイル命名規約
- 提案書・パーツ資料：`{客先英名}_{案件概要}_v{バージョン}.pptx`
  - 例：`Colliers_OMN_AI予測機能_ご提案_v0.1.pptx`
- 複数テーマを番号管理する場合：`{NN}_{Category}_{ThemeName}_v{バージョン}.pptx`
  - 例：`02_Equipment_ProcessAdjustment_v5.pptx`
- ROI Excel 等：`{NN}_{テーマ}_v{バージョン}.xlsx`
- バージョンは初期たたき台が `v0.1`、確定版に向けて `v0.2 → v1.0` と上げる

### 提案文の締め方
強い主張の後は柔軟性を残す一文で締める：
> なお、本優先度はあくまで弊社の特徴を踏まえたご参考であり、いずれにつきましても柔軟にご対応いたします。貴社のご判断・全体最適に応じてご相談ください。

### 複数資料の整合性
同一案件で複数資料を作る場合、数字・用語・前提を一貫させる（例：ROI Excel の運用保守費と運用費試算スライドの金額を食い違わせない）。

## 過去案件からの教訓

| 教訓 | 学んだ場面 |
|---|---|
| **pptxgenjs 出力は必ず STEP 3.5 でサニタイズする（`[Content_Types].xml` の幽霊 slideMaster Override 除去）** | 「開くたびに修復が必要」と指摘された。原因は pptxgenjs がスライド枚数ぶん実在しない `slideMaster{N}.xml` への `<Override>` を書き出すバグ（#1449）。`scripts/sanitize_pptx.mjs` で除去する。LibreOffice の PDF 化では検知できないため STEP 4 では気付けない |
| **スキル/example にない独自スライド・装飾を発明しない** | ネイビー一色＋マゼンタ下線の章扉を創作し指摘された。章扉は表紙と同じテクスチャ仕様 |
| **MAGENTA は対比・効果だけ。汎用アクセントにしない** | 矢印・章番号下線・「追加」帯・ロードマップで紫を多用し指摘された。NAVY が主色 |
| **CLIENT はスライド要素の色付けに使わない。入力カード・比較元帯・顧客側要素もすべて NAVY** | CLIENT を入力カードや「市場環境の変化」等の帯に使い指摘された。CLIENT は定数値であり色として使う色ではない |
| **カードを色分けしたいだけの理由で CLIENT・MAGENTA を使わない** | ネクストアクションカードに CLIENT/MAGENTA を割り当て指摘された。全て NAVY が正しい |
| **カード本文エリアの背景は WHITE のみ。薄色（CLIENT_LIGHT 等）は禁止** | CLIENT_LIGHT・MAGENTA_LIGHT をカード本文背景に使い指摘された |

LAYOUT.MARGIN : acc[i-1] + cws[i-1]); return acc; }, []);` |
